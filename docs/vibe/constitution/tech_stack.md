# Tech Stack Constitution

## 技术边界

1. Gin 仅可作为 Delivery / HTTP 适配器使用。
2. GORM 仅可存在于 Repository / Adapter 内部。
3. Redis、JWT、RBAC、缓存客户端、外部 SDK 等基础设施能力，只能通过 Adapter 暴露给上层 Port。
4. Usecase 与 Domain 不得直接依赖任何框架实现。

## 数据库结构的单一事实源

1. `cmd/db` 结合 `pkg/migrate` 是数据库结构变更的唯一标准入口。
2. 所有表结构新增、字段调整、索引变化、约束变化，都必须通过迁移脚本表达。
3. 运行期服务启动流程不得再承担 schema 变更职责。
4. 迁移脚本必须进入版本控制，并具备可审计、可追踪、可回滚的属性。
5. `scripts/migrations` 应作为数据库结构历史的唯一归档位置。

## 明确废弃项

以下链路已被正式废弃，不得继续扩散或复活：

1. 旧版 `initdb` 链路。
2. 运行时 `AutoMigrate`。
3. 独立于迁移系统之外维护的重复 schema 定义副本。

## 执行要求

1. 修改数据库结构时，先调整模型映射与适配层，再生成或编写迁移脚本。
2. 任何兼容壳都只能作为短期过渡，不得成为长期入口。
3. 文档、Makefile、CLI 示例与真实入口必须保持一致。
4. 命令名、目录名、仓库名与文档说法必须统一，禁止并存多套表述。

## 容器与依赖装配规范

1. 现有臃肿的 `internal/app` 必须持续拆分。
2. 顶层应用只应承担 Composition Root 职责。
3. 未来必须按模块组织依赖装配，例如 `User`、`Auth`、`Role`、`RBAC` 等。
4. 每个模块必须提供自己的 Provider 或 Wire 集合。
5. 顶层应用统一注册各模块，而不是手工编排每个业务对象的内部细节。
6. 模块内部依赖应尽量闭合在模块边界内。
